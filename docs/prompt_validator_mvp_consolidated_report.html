1. Objective & goals

Objective: Provide one persona-aware prompt quality service that scores prompts, lists issues, optionally rewrites them, persists results for analytics, and exposes the same capability through web UI, REST API, MCP, Microsoft Teams, and Slack.

Goals: Consistent evaluation against configurable personas; auditable source-of-truth configuration; measurable adoption via analytics; deployability on serverless hosting (Vercel); governance-oriented metadata on validations.

2. Business problem

Inconsistent prompt quality drives rework, higher LLM token use, and weak cross-role standards. Fragmented tools prevent a single audit trail and coaching loop.

3. Solution summary

A FastAPI application (app/main.py) exposes REST under /api/v1/*, serves a static web UI at /, implements MCP at POST /mcp (JSON-RPC 2.0) plus legacy /mcp/* helpers, handles Teams at POST /api/messages, and Slack at POST /api/v1/slack/validate. Core validation uses run_llm_validation with LLM providers (Anthropic / Groq) and optional static prescreen and fallbacks. Persistence uses MongoDB Atlas (default) or SQLite.

5. Workflow (high level)

  1. Ingress: prompt + persona_id (+ optional user_email, channel).
  2. Optional persona resolution when using baseline persona with email.
  3. LLM validation (and rewrite when auto_improve is true).
  4. Response: score, rating, strengths, issues, suggestions, improved prompt, dimensions.
  5. Background persistence: raw events, validation records, enriched events.
  6. Downstream: analytics and intelligence repositories.

9. Database & limitations

  • MongoDB: default production; connection via MONGODB_URI or Atlas-style env vars.
  • SQLite: local file prompt_validator.db for dev.
  • Limitations: dual code paths to maintain; Teams persona state in memory is not durable across instances; admin credentials in source are not enterprise-safe; permissive CORS in app settings.

Cost & budget — Anthropic and channels

Principle: LLM cost scales with validations and tokens, not with “channel” — Web, API, MCP, Teams, and Slack share the same validation engine. Platform costs (Vercel, MongoDB, Azure Bot) are usually secondary to LLM at moderate scale.

Pricing assumption (verify on Anthropic): illustrative Sonnet-class rates ~$3 / M input and ~$15 / M output tokens. Each validation often uses up to two LLM calls (evaluate + rewrite) when auto_improve is true; prescreen and cache can reduce calls.

Daily usage limits

  • Product quotas: define per-user and org caps at API gateway if required.
  • Provider limits: Anthropic tier limits (RPM/TPM/spend) in Anthropic console.
  • Code: in-memory cache and static prescreen reduce redundant LLM spend.

Governance & security

Data governance (what exists)

  • Governance metadata: delivery channel, scores, LLM provider/model; masked email hash fields in governance payload (app/services/data_strategy.py).
  • Source-of-truth document references.
  • Compliance checklist endpoint for Day-2 style verification.

Gaps

  • Full prompt text may be stored and sent to the LLM — no automatic DLP redaction of secrets in prompt body by default.
  • Some routes may be exposed without API key depending on deployment.
  • Admin: hardcoded credentials pattern in routes — replace for production.
  • CORS configured permissively — tighten for production.

Prompt injection

Yes — relevant. User-supplied text is embedded in LLM requests. System prompts constrain behavior but do not deterministically block adversarial instructions. Plan mitigations: gateway auth, quotas, retention, monitoring, residual LLM risk acceptance.

Final conclusion — summary report

Summary: The Prompt Validator MVP delivers a unified prompt-quality engine with multiple channels (web, REST, MCP, Teams, Slack), persona-based evaluation, persistence for analytics, and governance-oriented metadata. It is well suited as a pilot or CoE showcase.

Production readiness: requires hardening — authentication/authorization on sensitive routes, secrets and admin flows removed from code, CORS and network policy, retention and residency alignment, abuse and spend controls, and explicit handling of residual LLM risks including prompt injection.

Bottom line: Strong functional MVP for controlled rollout; invest in security and governance before org-wide mandatory use or regulated data.
