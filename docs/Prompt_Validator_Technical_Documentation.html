Prompt Validator — Technical Documentation

Version 1.0  |  Stack: FastAPI · Python 3.10+ · Groq LLM · MongoDB · React (Frontend)  |  Last Updated: April 2026

The Prompt Validator is a persona-aware, enterprise-grade prompt quality scoring and improvement engine built at Infovision CoE AI/GenAI Practice. It evaluates user prompts against a curated source-of-truth framework, scores them across multiple dimensions, and automatically rewrites them into structured, guideline-compliant improved prompts using a multi-step LLM pipeline.

Core Value Proposition Users submit a raw prompt → the engine evaluates quality across 7+ weighted dimensions → assigns a 0–100 score → identifies gaps → rewrites the prompt into a fully structured, persona-specific improved version — all in under 5 seconds.

Validation Pipeline — End-to-End Flow

1

Request Received — POST /api/v1/validate

FastAPI validates the request schema (Pydantic), checks the API key header, resolves the persona_id (auto-mapped from user_email if provided), normalises the delivery channel, then calls run_llm_validation(prompt_text, persona_id, auto_improve).

2

Cache Lookup ~0 ms

A SHA-256 key is computed from (persona_id + auto_improve + prompt_text). If an identical request was made within the last PROMPT_CACHE_TTL_SECONDS (default 600 s / 10 min), the cached result is returned immediately — no LLM or rules engine called.

cache_key = SHA256(f"{persona_id}:{auto_improve}:{prompt_text.strip()}")
if key in cache and time.monotonic() < expire_ts:
    return cached_result
3

Static Pre-screen ~5–15 ms

The lightweight keyword-based rules engine scores the prompt statically. If static_score ≥ STATIC_PRESCREEN_THRESHOLD (default 70), the LLM evaluation call is skipped entirely. This saves ~2,000–4,000 tokens per request for prompts that already score well.

if static_score >= STATIC_PRESCREEN_THRESHOLD:
    scoring_mode = "static_prescreen"
    skip LLM eval  # rewrite still runs if auto_improve=True
4

LLM Semantic Evaluation ~2–5 s

If the static pre-screen doesn't short-circuit, the prompt is sent to the LLM for deep semantic evaluation. The LLM receives a structured payload containing the persona dimensions, guideline rules, and source-of-truth document reference. It returns dimension pass/fail results, issues, strengths, and guideline checks in a strict JSON schema.

The server then recomputes the final score from the LLM's dimension pass/fail results (overriding the LLM's own semantic_score to guarantee mathematical consistency).

5

Score Computation

base_score   = (sum_passed_weights / sum_all_weights) × 100
penalty      = min(failed_guideline_checks × penalty_per_miss, penalty_cap)
final_score  = max(0, base_score − penalty)

Zero-weight dimensions (TRACEABILITY, BUSINESS RELEVANCE) are filtered out of the response to prevent confusing 0/0 display.

6

Suggestion Derivation

The suggestion_engine maps identified issues to persona-specific, actionable suggestions. For example, a missing "edge case" issue for persona_1 (Dev+QA) maps to: "[QA] Request explicit edge coverage: boundary values, null inputs, timeout, and error states."

These suggestions are also passed to the LLM rewrite as guided gaps to resolve.

7

LLM Prompt Rewrite ~2–4 s

If auto_improve=true, the LLM is called a second time to rewrite the original prompt into a fully structured, section-based improved prompt. This uses a 3-step system prompt: Fact Extraction → Contradiction Detection → Write Improved Prompt. The output is a JSON object containing improved_prompt, applied_guidelines, and unresolved_gaps.

8

Result Assembly & Cache Store

All results are assembled into a ValidateResponse, history is saved to MongoDB, analytics events are captured, and the result is stored in the in-memory cache for future identical requests.

9

Response Returned Total: ~4–8 s end-to-end

The full ValidateResponse JSON is returned to the client — score, rating, dimension breakdown, issues, suggestions, improved prompt, guideline evaluation, and LLM metadata.

Static Rules Engine

The rules engine (app/services/rules_engine.py) scores prompts without any LLM call using two mechanisms:

1. Dimension Keyword Checks

Each persona defines a keyword_checks map in persona_criteria_source_truth.json. The engine checks whether any of the dimension's keywords are present in the prompt text.

2. Global Guideline Checks

A set of global checks from prompt_guidelines.json are evaluated against the prompt:

Check IDDescriptionPenalty if Failed
clear_direct_actionStarts with an action verb or direct instruction3 pts
context_groundingProvides source material, audience, or purpose context3 pts
output_format_controlDeclares expected output format (list/JSON/table/code)3 pts
constraints_presentIncludes constraints: tone, length, scope, exclusions3 pts
structured_for_complexityUses structure (steps/sections/tags) for complex requests3 pts
factual_grounding_for_analysisRequires grounded/cited output for analytical tasks3 pts
Penalty Cap Total guideline penalty is capped at strict_penalty_cap (default: 15 pts) regardless of how many checks fail.

LLM Evaluation — What We Send & Receive

System Prompt (Role & Rules)

The LLM is instructed to act as an "enterprise prompt-quality auditor" operating strictly against the provided source-of-truth framework. Key rules injected:

  • Evaluate ONLY the dimensions listed in persona_dimensions — do not add or remove any
  • Mark a dimension passed=true ONLY when the prompt explicitly contains keyword evidence AND satisfies at least one criterion
  • Compute score mathematically: (sum_passed_weights / sum_all_weights) × 100 − guideline_penalty
  • List ONLY issues that are actually missing — max 6 unique issues
  • No hallucination — never credit the prompt for content not present in the text

Server-Side Score Recomputation

Why We Recompute the Score LLMs frequently return a semantic_score inconsistent with their own dimension evaluations (e.g., all dimensions pass but score is 82 instead of ≥85). The server always recomputes the final score from dimension pass/fail results to guarantee mathematical correctness.

LLM Prompt Rewrite — 3-Step Process

When auto_improve=true, a second LLM call rewrites the original prompt using a strict 3-step system prompt:

1

STEP 1 — Extract Immutable Facts (Silent)

Before writing anything, the LLM scans the original prompt and extracts every explicit fact: programming language, frameworks, versions, endpoint paths, field names, HTTP status codes, output format directives, edge cases, test requirements. These facts are immutable — they must be carried forward unchanged into the improved prompt.

2

STEP 2 — Detect & Flag Contradictions

The LLM checks for 5 contradiction patterns:

  • C1 — Framework ↔ Language mismatch (e.g., FastAPI + Java)
  • C2 — Conflicting output formats (e.g., JSON AND Markdown table)
  • C3 — Conflicting scope (include auth AND auth is out-of-scope)
  • C4 — "Latest version" ambiguity
  • C5 — Missing critical fields (for prompts <20 words)

Contradictions are flagged in unresolved_gaps — never silently resolved.

3

STEP 3 — Write Improved Prompt

The improved prompt is written using persona-specific section headings in a fixed order. Each section is populated from Step 1 facts expanded into clear expert sentences. A specific expert role is derived from the task domain (e.g., "You are a Senior Java Engineer specialising in MongoDB integration").

Rate Limit Handling

Free tier limits:
  ~30 requests/minute
  ~14,400 tokens/minute  (for llama-3.3-70b-versatile)

Retry logic:
  max_retries = 3
  On 429:
    wait = parse Retry-After header (or exponential backoff)
    if wait > 10 seconds → raise GroqRateLimitError IMMEDIATELY
      (prevents 60s hang; triggers static fallback in <1s)
    else → sleep(wait) and retry

Process-wide cooldown:
    _next_groq_request_at = monotonic() + wait_seconds
    Next request checks cooldown BEFORE sending:
      if cooldown > 10s → raise immediately (no waiting)

Client-Side Pacing

GROQ_REQUESTS_PER_MINUTE = 25
min_interval = 60 / 25 = 2.4 seconds between requests
Enforced via _enforce_client_side_pacing() before every Groq call

Anthropic LLM Integration

Anthropic (claude-sonnet-4-6) is the secondary LLM provider. It activates when:

  • LLM_PROVIDER=anthropic — explicitly configured
  • LLM_PROVIDER=auto — Groq is unavailable or unconfigured
  • Groq returns 429 AND LLM_PROVIDER=auto — automatic failover to Anthropic
Production Recommendation For production deployments, set LLM_PROVIDER=anthropic. Anthropic has no per-minute token limits comparable to Groq's free tier and produces consistently higher-quality evaluations and rewrites.

Scoring Formula

Step 1 — Dimension base score:
  base_score = (Σ passed_dimension_weights / Σ all_dimension_weights) × 100

Step 2 — Guideline penalty:
  penalty = min(failed_guideline_check_count × penalty_per_miss, penalty_cap)
  Default: 3 pts per miss, capped at 15 pts

Step 3 — Final score:
  final_score = max(0, base_score − penalty)

Rating thresholds:
  ≥ 85  →  Excellent
  ≥ 70  →  Good
  ≥ 50  →  Needs Improvement
  < 50  →  Poor
