INFOVISION
Enterprise Prompt Validator
Phased Development Strategy & Executive Proposal
One engine. Five personas. Four delivery channels. One consistent standard.
3,000 Consultants · Global Delivery · Multiple Engagement Models
Version 2.0  |  March 2026  |  Confidential — Internal Strategic Document
Executive Summary
Infovision operates across thousands of consultants, multiple geographies, and a wide range of client environments. As AI tools become embedded in daily workflows, the quality of the prompts our teams write directly influences the quality of the outputs they receive — and, by extension, the quality of the work we deliver to clients.
This document presents the strategic plan for building the Infovision Enterprise Prompt Validator: a system that evaluates, scores, and improves prompts written by every consultant, across every role. It is built as a centralized intelligence platform and delivered progressively through multiple access channels — each one extending the same capabilities to a new environment or working context.
The system is structured around a layered API architecture built in Phase 1: one universal enterprise-wide endpoint serving all employees, plus dedicated persona-specific endpoints calibrated for each role cluster. All endpoints share the same core intelligence. All phases connect to the same backend. No logic is ever duplicated.
Each phase builds directly on the previous one. The backend grows more refined with use. Each new channel adds reach without adding complexity to the core.
Context and Problem Statement
The Challenge We Are Solving
Infovision consultants use AI tools every day across every function. Developers write prompts to generate code. QA engineers write prompts to produce test cases. Business analysts write prompts to extract requirements from documents. Product owners write prompts to draft user stories. Support staff write prompts to compose customer responses. Project managers write prompts to generate status reports.
Across all of these roles and use cases, the quality of the prompt is the primary determinant of the quality of the output. Yet today, Infovision has no consistent standard for what a good prompt looks like, no feedback mechanism to help consultants improve, and no organizational visibility into prompt quality across teams, roles, or engagements.
The result is predictable: inconsistent AI output quality across every function, missed value from AI tools, unnecessary rework, and no structured path to improvement — for any persona.
The Five Personas This System Serves
The Prompt Validator is designed to serve every consultant who writes a prompt. This covers five organizational persona clusters, each with distinct prompting needs, quality dimensions, and scoring criteria:
All five personas are served by the same backend engine from Phase 1 onward. The scoring criteria and quality dimensions differ per persona. The intelligence infrastructure does not.
The Enterprise Reality We Are Designing For
Our consultants operate across deeply varied working conditions that make a one-size-fits-all delivery model impractical:
This is why the system is delivered through multiple channels across four phases — not because different personas require different phases, but because different environments require different access models. All personas must be reachable through every channel that works in their environment.
System Architecture: One Engine, All Personas, Many Access Points
The most important architectural decision in this system is the separation of intelligence from delivery. The validation logic — the scoring dimensions, the persona-calibrated evaluation criteria, the autofix generation, the improvement explanations — lives in one place: the Phase 1 backend engine. Every phase that follows is simply a new way to reach that same intelligence.
The Layered API Architecture
Phase 1 builds the backend engine with a layered API structure that serves all five personas simultaneously. This architecture has two tiers:
Every API endpoint in this structure connects to the same core intelligence. The persona-specific endpoints apply a different scoring profile — different weights, different dimension priorities, different autofix guidance — but the underlying engine, the data model, and the storage layer are identical across all of them.
When a Phase 3 chatbot serves a Support Staff member, it calls the Support Staff persona API. When it serves a Business Analyst, it calls the BA/PO persona API. The chatbot does not change. The backend applies the right evaluation. This is true for every channel in every phase.
How Phases Relate to Personas
Every cell in this grid is checkmarked because that is the design intent. No phase is scoped to a subset of personas. No persona is deferred to a later phase. The layered API architecture in Phase 1 ensures that all scoring profiles are available from the start. Each subsequent phase simply opens a new door into the same room.
Phase 1 is the foundation and the substance of the entire system. It is not a stepping stone to a future product — it is the product itself. Everything built in Phases 2, 3, and 4 is simply a different way of reaching what Phase 1 creates.
The defining characteristic of Phase 1 is completeness: the backend engine built here serves all five organizational personas from the moment it is deployed. This is not a phased persona rollout — all persona scoring profiles, all API endpoints, and all evaluation dimensions are built and operational in Phase 1. What changes in later phases is the channel through which people access this intelligence, not the intelligence itself.
Phase 1 has three primary commitments: establish Infovision's formal, measurable definition of prompt quality for every persona; build the data collection and storage infrastructure that will power organizational analytics; and create an API architecture that is extensible, stable, and ready to support every channel from Phase 2 onward.
Core Validation Intelligence
Prompt ingestion and persona context routing: the engine accepts a prompt along with persona and context metadata, and routes it to the correct scoring profile
Universal baseline scoring: every prompt is evaluated against the organization-wide baseline dimensions that apply to all employees regardless of role
Persona-specific scoring layers: each prompt is additionally evaluated against the elevated dimensions calibrated for the declared persona — with different weights, different priorities, and different quality thresholds per role cluster
Issue identification: clear articulation of what weakens the prompt and why it reduces AI output quality, expressed in language appropriate for that persona
Improvement suggestions: specific, actionable recommendations written for the persona's context and typical use case
Auto-fix generation: an automatically rewritten version of the original prompt that addresses all identified gaps, calibrated to the persona's prompting standards
Side-by-side comparison: a structured output presenting the original prompt, the improved prompt, and a plain-language bullet-point explanation of each change made
Layered API Architecture
Universal enterprise API endpoint: accepts any Infovision employee's prompt and applies baseline scoring — the entry point for all general usage and the foundation for all persona-specific endpoints
Developer + QA persona API: applies elevated scoring for technical precision, edge-case coverage, testability, constraint definition, and code-specific output standards
PM / TPM persona API: applies elevated scoring for output format, reproducibility, context completeness, traceability, and sprint-and-report-specific quality dimensions
BA + Product Owner persona API: applies elevated scoring for retrieval grounding, accuracy, business relevance, prioritization clarity, and document-synthesis standards
Support Staff persona API: applies elevated scoring for tone and user empathy control, compliance, actionability, and customer-facing output quality
All five API endpoints share the same core engine, the same data model, and the same storage layer — they differ only in their scoring profiles
Data Collection and Organizational Intelligence Framework
Prompt storage: every prompt evaluated is captured with full context — persona, channel, timestamp, score breakdown, identified issues, suggestions, and autofix output
Scoring metadata: dimension-level scores are stored individually, enabling analysis of where specific personas or teams consistently underperform on specific quality criteria
Improvement tracking: before-and-after prompt pairs are stored, creating a longitudinal record of how prompt quality evolves across the organization over time
Analytics-ready data model: the storage structure is designed from day one to support organizational dashboards, team-level benchmarking, and training program design
Channel attribution: every record captures which access channel submitted the prompt, enabling cross-channel performance comparison in later phases
Extensibility and Channel-Readiness
Stable, documented API contracts: the API surface is designed with future channels in mind — clean, predictable, and consistent, so that Phases 2, 3, and 4 can integrate without requiring backend changes
Structured response format: all API responses follow a consistent schema that can be consumed by chatbots, LLM tools, IDE plugins, and web interfaces equally without transformation
Versioning and backward compatibility: the API is built to accommodate future updates to scoring models without breaking existing channel integrations
Environment configuration: the engine supports different operational modes — direct API, proxy-mediated, and limited mode — to accommodate the range of enterprise network conditions it will encounter across channels
Phase 1 is governed by four non-negotiable priorities:
Infovision has, for the first time, a formal and measurable definition of prompt quality — one that applies consistently across all five organizational personas
All five persona scoring profiles are available immediately, meaning any channel built in later phases can serve the full organization from its launch day
The data collection framework begins accumulating organizational intelligence from the first prompt evaluated, regardless of which channel submitted it
The stable API architecture means Phase 2, 3, and 4 teams can begin integration planning in parallel, reducing the total time from Phase 1 completion to full system deployment
Leadership and stakeholders gain confidence in the system's institutional design — it is not a tool with an unknown roadmap, but an engine with a clear architecture that will serve the organization for years
Phase 1 has no dependencies — it is the origin of the system. It is also the only phase with no predecessor. All three subsequent phases depend on Phase 1 being complete, stable, and fully persona-capable before they begin integration:
Phase 2 calls Phase 1's layered API to expose all five persona endpoints as LLM-native tools — all personas available via MCP tooling
Phase 3 uses Phase 1's API to power chatbot interactions across Teams, Slack, and Discord — all personas accessible through chat
Phase 4 connects to Phase 1's API for inline IDE validation — all personas served through development environment plugins
Critically: no subsequent phase introduces new validation logic. If a scoring dimension changes, if a new persona is added, or if autofix quality is improved, that change happens once in Phase 1 and is automatically reflected across all channels.
Phase 1 has no direct user interface — this is intentional. Building a UI in Phase 1 would create a dependency that constrains channel-agnostic design. The interface comes later, through the appropriate channels.
Persona completeness requires upfront investment: building all five scoring profiles in Phase 1 is more work than building one. This investment pays back immediately when each subsequent channel launches with full organizational coverage rather than partial coverage.
The data collection framework must be designed before volume exists, not in response to it — architectural decisions made under volume pressure are consistently worse than those made with care upfront.
API stability is a Phase 1 responsibility: changes to API contracts after Phase 2 begins integration create cascading rework. The API surface must be treated as a commitment, not a prototype.
Phase 2 does not add new intelligence. It adds a new way for every persona to reach the intelligence built in Phase 1.
The Model Context Protocol (MCP) is an emerging standard that allows LLM systems to call external tools and services as part of their reasoning and response generation. By exposing the Prompt Validator as an MCP-compatible tool in Phase 2, all five persona endpoints become callable by any LLM-driven workflow — automatically, without requiring a separate submission step by the user.
This phase matters for every persona, not just technical ones. A business analyst building an AI-assisted requirements pipeline, a product owner using an AI workflow to generate user stories, a support manager using an LLM-powered ticket system — all of them can have prompt validation embedded in their AI workflow through the MCP layer, calling the appropriate persona endpoint transparently in the background.
MCP tool definitions for all five persona endpoints: each persona API from Phase 1 is formally defined as a callable MCP tool — with its own schema, input parameters, and structured output format — so LLM systems can invoke the correct persona validator as part of their pipeline
Universal tool entry point: a single MCP tool definition that routes to the correct persona endpoint based on context declared in the calling workflow, enabling workflows that span multiple personas
Structured LLM-consumable responses: all validation outputs (score, issues, suggestions, autofix) are formatted for programmatic consumption by LLM orchestrators, not just human readers
Tool registration across relevant LLM environments: the persona-aware validator tools are made available in the AI assistant configurations, copilot integrations, and orchestration frameworks used across Infovision
Composable pipeline integration: the validator tools are designed to work as components in multi-step AI pipelines — for example, validate a prompt, rewrite it, then pass the improved version to the next tool in the workflow
Persona-aware tool exposure: each MCP tool definition must correctly route to the corresponding Phase 1 persona API, ensuring that LLM workflows trigger the right scoring profile automatically
Standardized interoperability: MCP tool definitions must conform to LLM tool-calling conventions so they are compatible with the AI orchestration frameworks already in use across Infovision
Minimal latency: tool invocation must add negligible processing time to workflows — validation should be fast enough that embedding it in an AI pipeline does not meaningfully slow the overall experience
Consistent output: MCP responses must be identical in scoring substance to what the API returns in any other channel — no logic divergence, no persona scoring shortcuts
All five personas gain access to automated, inline prompt validation within their AI-driven workflows — without any manual submission step
Prompt quality is enforced at the pipeline level, not just as a manual check — for every persona whose work flows through an LLM system, validation becomes part of the process rather than an optional add-on
Infovision teams building AI-powered internal tools can incorporate persona-aware validation as a native component, without building scoring logic from scratch
Phase 2 data flows into Phase 1's storage layer, adding a new stream of organizational prompt quality intelligence alongside direct API usage
Depends on Phase 1 completely: Phase 2 exposes what Phase 1 built. Without a complete, persona-aware Phase 1 backend, there is nothing useful to expose through MCP tooling.
Does not depend on Phase 3 or Phase 4: Phase 2 can be developed in parallel with or ahead of chat and IDE integrations, as its target environment (LLM orchestration) is independent of messaging platforms and development environments
Supports Phase 4: IDE plugin integrations that include copilot-style AI assistance can leverage Phase 2's MCP tool definitions to embed validation more deeply into AI-native development workflows
MCP tool calling requires that the host LLM system supports tool use — this is available in most modern AI platforms but may be restricted in some client-controlled environments; Phase 2 is a progressive enhancement, not a prerequisite for Phase 3 or Phase 4
Persona routing in automated workflows requires that the workflow system correctly declares the persona context when calling the validator tool — design guidance for workflow builders must accompany Phase 2 deployment
Data routing through MCP must respect the same data sensitivity considerations as direct API calls — the tool must support environment-specific configuration to prevent data leaving restricted networks for any persona

Phase 3 exists where it does in the roadmap because it requires Phase 1 to be fully operational — including all persona endpoints — and because chat-based interfaces are one of the most universally accessible channels across Infovision's varied working environments. Microsoft Teams bots, in particular, often function even on client-provided machines where external websites and installable plugins are blocked.

End-to-End System Flow
The Prompt Validator is not four separate systems built in sequence. It is one system — the Phase 1 backend engine — that is progressively made accessible through four delivery channels, each reaching the same five personas through a different access model.
How the System Evolves Across Phases
Phase 1 creates the complete organizational definition of prompt quality for all five personas. From the moment Phase 1 is operational, every scoring profile, every autofix model, and every persona API endpoint exists and is ready. The system knows what a high-quality prompt looks like for a developer, a business analyst, a support agent, a project manager, and a product owner.
Phase 2 takes that complete intelligence and makes it callable by LLM systems and AI orchestration workflows — for all five personas. A requirements extraction AI workflow can call the BA persona validator automatically. A test generation pipeline can call the QA persona validator inline. An AI-powered reporting tool can validate a PM's prompt before generating the status report.
Phase 3 takes the same complete intelligence and makes it accessible through conversational interfaces. A support agent in a Teams channel can validate a customer reply prompt and receive tone and compliance feedback without leaving the messaging platform. A product owner in Slack can validate a user story prompt and receive actionability and AC-completeness feedback. Every persona, through one consistent channel.
Phase 4 takes the same complete intelligence into the development and knowledge-work environment. A developer validates a code generation prompt inline before it reaches the AI copilot. A business analyst validates a requirements extraction prompt while drafting in an editing environment. The validator is present at the moment of work, for every persona that uses an IDE-style tool.
The Universal Data Loop
Every interaction across every channel in every phase flows back into Phase 1's storage layer. A developer using Phase 4, a support agent using Phase 3, a business analyst using Phase 2 — all of their prompt evaluations are captured with full persona and channel attribution in the same data model.
This creates an organizational dataset that is uniquely cross-functional. Leadership can see prompt quality trends not just for developers, but for all five personas — across all four channels — in a single view. Questions like 'which persona has the highest autofix adoption rate?' or 'which channel produces the most complete business analyst prompts?' become answerable from the data the system naturally generates.
All Channels, All Personas: The Complete Picture
Core Design Principles
The following principles guide every decision in the design and phasing of the Prompt Validator. They are not aspirational — they are active design constraints that have shaped the architecture, the phase order, and every capability decision.
Why This Approach Works
It Serves Every Persona at Every Step
A common failure mode in enterprise platform design is solving for the most visible or technically sophisticated users first, then expanding to other personas over time. This approach rejects that pattern entirely. By building all five persona scoring profiles into Phase 1, Infovision ensures that from the moment the first channel goes live, every consultant can access persona-calibrated validation — not a generic, one-size-fits-all evaluation that serves no one particularly well.
A support agent using Phase 3 on day one gets tone and compliance scoring, not a developer scoring profile applied incorrectly to a customer reply prompt. A business analyst using Phase 2 gets retrieval grounding evaluation, not a generic clarity check. The persona-specific value is present from the first interaction, in every channel.
It Matches Our Enterprise Reality
Infovision's working environment is not a controlled lab. We work on client machines with varying levels of restriction, in VPN-controlled networks, and in accounts with strict data governance requirements. No single channel works everywhere for all five personas.
The multi-channel strategy exists precisely to solve this. If an IDE plugin is blocked on a client machine, the same developer can use the Teams bot. If the Teams bot is unavailable, the same business analyst can submit via direct API call. If network access is restricted, the universal baseline runs locally. Every persona always has at least one working path to the same evaluation intelligence.
It Avoids Duplication and Its Consequences
When scoring logic lives in multiple places — one version in the chatbot, another in the plugin, a third in the LLM tool — they diverge. Updates are applied inconsistently. Different personas receive different quality evaluations for identical prompts depending on how they access the system. Trust in the standard erodes.
Because all channels call Phase 1, there is one scoring model. When the BA/PO persona scoring dimensions are updated, every channel — the chatbot, the plugin, the MCP tool — reflects the change immediately without any additional work. Organizational confidence in the standard is maintained because the standard is, in practice, singular.
It Delivers Value at Each Phase, for All Personas
It Creates Organizational Learning Across All Personas
Because all personas contribute to the same data layer, the organizational analytics are uniquely rich. Leadership can compare prompt quality maturity across roles — not just within a single persona group. They can identify whether the BA team has stronger context completeness than the PM team, whether support staff have improved their tone calibration after a training program, or whether developer prompts submitted through the IDE plugin score higher than those submitted through chat.
This cross-persona, cross-channel view of prompt quality is only possible because the architecture routes all personas through the same engine and stores all interactions in the same data model. It cannot be recreated by aggregating data from separate, persona-specific tools.
Final Summary
Phase 1 builds the complete backend: five persona scoring profiles, a layered API architecture, an autofix engine, an improvement explanation system, and a data collection framework — all operational before the first channel goes live. This is the source of truth for prompt quality at Infovision. It does not grow in persona coverage across phases. It grows in reach.
Phases 2, 3, and 4 are delivery channels. Each one extends the same backend capabilities — serving all five personas — to a new environment and working context. No phase introduces new validation logic. No phase is restricted to a subset of personas. The only thing that changes across phases is how people access what Phase 1 already knows.
This architecture works because the intelligence is singular and the access is plural. The Prompt Validator is not four tools solving the same problem four ways. It is one solution reaching every corner of the organization — every persona, every environment, every phase — through the channel that works best in each context.

Quality Dimension Importance
Clarity: CRITICAL — applies universally; unclear prompts produce poor output regardless of role or tool
Context Completeness: HIGH — all prompts benefit from context; output quality directly correlates with input quality
Specificity: HIGH — vague prompts produce generic, non-actionable output across every use case
Output Format: HIGH — declaring the expected format (list, paragraph, table) dramatically improves usability
Ambiguity Reduction: HIGH — ambiguous task framing causes rework across all Infovision functions
Constraint Definition: MEDIUM-HIGH — all employees work within limits (length, tone, scope) that should always be stated
Actionability: MEDIUM-HIGH — outputs should enable next actions, not just surface information
Accuracy Orientation: MEDIUM — grounding prompts in provided content reduces hallucination risk for all roles
Clear task framing — Begin every prompt with an explicit action verb: Summarize / Draft / Extract / Analyze / Explain / Create. Never omit the action.
Explicit context — Always provide: who you are, what the document or situation is, and what the output will be used for. Context is the single biggest driver of output quality.
Output format declaration — Always specify how you want output formatted: bullet list, numbered steps, table, paragraph, or specific template. Without this, output structure is unpredictable.
Constraint definition — State all relevant limits before submitting: tone (formal/casual), word count, audience level (technical/non-technical), and scope (what to include and exclude).
Audience specification — Always declare who the output is for: internal team, external client, executive leadership, or end customer. Output language and depth must match the reader.
Avoid open-ended prompting for professional outputs — 'Tell me about X' produces generic results. 'Summarize the top 3 risks in this document for a client executive in 5 bullet points' produces enterprise value.
Prefer deterministic formatting for recurring outputs — Status updates, summaries, and reports benefit from fixed format requests. Always ask for consistent structure across recurring prompt types.
Use examples as anchors — For non-obvious output types, include a 1–2 sentence example of what good output looks like. This eliminates format ambiguity.
Refine iteratively, not from scratch — If output is close but not right, add a constraint or example in a follow-up prompt rather than rewriting from zero.
Define acceptance criteria before submitting — Ask yourself: what does a good output look like for this prompt? If you cannot answer, the prompt is not ready.
What the Validator Must Check
Does the prompt begin with an explicit action verb (Summarize / Draft / Analyze / Explain / Create)?
Is any context provided about the source material, situation, or purpose of the output?
Is the expected output type or format declared (list, paragraph, table, template)?
Is the prompt open-ended with no constraints on scope, tone, length, or audience?
Is the intended audience or downstream use of the output mentioned?
Penalty Triggers
Zero context provided — HIGH penalty (AI has no grounding; output will be hallucinated or generic)
No output format declaration — MEDIUM penalty (output structure will be unpredictable and inconsistent)
Completely open-ended prompt with no constraints — HIGH penalty (violates the core baseline rule)
Missing action verb — MEDIUM penalty (AI may misinterpret intent and produce the wrong output type)
Single-word or single-phrase prompt with no context — HIGH penalty (insufficient for any professional output)
